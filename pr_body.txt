What: Implement strict keyboard focus trapping for overlay components (Dialog and Drawer).
Why: Native `<dialog>` elements handle focus well on their own, but since we manually intercept closure events (like the `Escape` key) to orchestrate Svelte 5 out-transitions, we risk focus leakage. Implementing a dedicated `FocusTrap` ensures absolute compliance with WCAG 2.1.2 (No Keyboard Trap).
DESIGN.md sections cited: "2. Accessibility Floor".
DESIGN.md sections rewritten: Added strict focus trapping mandate to the "Accessibility Floor" section, as the previous text was too vague regarding modal mechanics.
Roadmap items advanced: epic-v151-visual-governance, epic-bdd5-spec-first, tactical-9j0k-complex-organisms (task-7w8x-focus-trap-overlays).
Decisions: Built a generic `FocusTrap.svelte` wrapper utility using a fast `onMount` listener instead of a Svelte action.
Trade-offs: Svelte wrapper components slightly increase DOM nesting (`style="display: contents;"`), but provide a much cleaner API for Svelte 5 snippets than `use:action` directives for entire DOM subtrees.
Test evidence: Added Gherkin feature and full Playwright E2E coverage in `e2e/dialog-drawer-focus.test.ts` (passing). All existing unit tests pass.
