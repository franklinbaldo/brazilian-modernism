**What I did:**
- Introduced dedicated semantic CSS variables (`--border-input` and `--focus-ring`) in `global.css`.
- Configured these variables to map to high-contrast neutral tokens (`--concreto-60`, `--concreto-80`) in the `.dark-mode` scope.
- Refactored `TextInput`, `Select`, `MultiSelect`, `Checkbox`, `Radio`, `Switch`, `FileInput`, `DatePicker`, `TimePicker`, `Combobox`, `FilterBar`, and `SearchBar` components to use the new tokens instead of generic `--border` and `--accent`.
- Authored a Gherkin feature file `features/dark-mode-form-contrast.feature` to document the expected accessible behavior.
- Expanded the Playwright suite `e2e/dark-mode-form-contrast.spec.ts` to programmatically measure and verify relative luminance contrast (ratio >= 3.0) for both standard form boundaries and focused states.

**Why:**
- To ensure WCAG AA non-text contrast compliance for form elements in dark mode. Previously, relying on the vibrant saturated primary colors or the low-contrast structural borders for form outlines led to contrast failures when the background was inverted. Decoupling the form tokens solves this while preserving the vibrant identity markers for other components.

**Decisions made:**
- Opted to use explicit overrides via semantic CSS tokens (`--border-input` and `--focus-ring`) rather than invasive JavaScript state tracking.

**Trade-offs:**
- None. This maintains backward compatibility and strengthens the accessibility foundation of the design system.

**Test evidence:**
- Playwright tests in `e2e/dark-mode-form-contrast.spec.ts` pass, asserting relative luminance ratios >= 3.0.
- `npx vitest run` passes across all existing component suites.
- `npm run build` succeeds without regressions.
