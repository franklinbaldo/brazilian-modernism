**What**
Implemented WCAG 2.2.1 (Timing Adjustable) compliance for the `Notification` component by introducing an auto-dismiss `timeout` prop that intelligently pauses when the user hovers over or focuses the notification, and resumes when the interaction ends. Also added comprehensive MDX documentation and E2E tests for the component.

**Why**
Notifications are asynchronous, application-level feedback. When a notification automatically disappears (auto-dismisses), it risks violating WCAG 2.2.1 if the user isn't given enough time to read it or a mechanism to pause the timer. Implementing pause-on-hover and pause-on-focus ensures the component remains accessible and considerate of user interaction rhythms.

**Decisions**
- Chose Svelte 5 `$state` and `$effect` to manage the countdown timer. This allows programmatic calculation of `remainingTime` and clean teardown on unmount, avoiding memory leaks.
- Added `tabindex="-1"` to the `.br-notification` wrapper to allow the element to natively capture `focusin` and `focusout` events, rather than relying on complex event bus logic.
- E2E tests use `.locator('.br-notification', { hasText: '...' })` rather than exact string matching, preventing flakiness caused by MDX compiling artifacts.

**Trade-offs**
- Managing intervals/timeouts natively in Svelte `$effect` blocks is slightly more verbose than using a dedicated animation library, but it adheres to our "restraint as a feature" doctrine by keeping dependencies zero and the component size extremely small.
