**What**
Implemented validation states for data entry components (`TextInput` and `Select`) adhering to the Curva & Concreto aesthetic. This ensures unmistakable visual feedback when inputs are invalid.

**Why**
To provide clear, accessible form validation to users while respecting the overarching design tokens set forth by the system.

**Decisions**
- Chose to manage the visual invalid states directly via CSS classes (`.invalid` and `.valid`) reacting to the `invalid`/`valid` component props (or via the Svelte `cobogo-form-field` context).
- Updated `TextInput` to explicitly transition `color` and `background-color` and apply `--vermelho` or `--verde` accordingly.
- Updated `Select` to apply the same text/background token variables, and importantly, dynamically swapped out the base64 encoded dropdown SVG arrow to match the appropriate semantic color (`#C8472E` for invalid, `#2E6B4A` for valid).

**Trade-offs**
- Modifying the base64 SVG directly within `Select.svelte` CSS creates slight duplication, but avoids importing heavier external icon handling inside base form components.

**Test Evidence**
- Added BDD feature spec: `features/forms/validation.feature`.
- Expanded Vitest coverage in `TextInput.test.ts` and `Select.test.ts` to assert that invalid and valid styling classes are correctly applied and that invalid prop takes precedence over the valid prop.
- Resolved an unrelated flaky E2E `dark-mode` test by adding a brief wait to ensure CSS transitions finish evaluating.
- Ran all tests: 123 unit tests pass, 17 E2E tests pass.

**DESIGN.md sections rewritten**
- `Form Organisms` -> renamed to `Form Atoms & Organisms`.
- Added a `Validation States` entry clearly establishing the token usage (`--vermelho-soft`, `--vermelho`, `--verde-soft`, `--verde`) for component feedback.
