**What I did:**
Implemented an `onkeydown` handler within `Tooltip.svelte` to listen for the `Escape` key and dismiss the tooltip without altering the user's focus. Added a `.feature` spec and comprehensive Vitest unit tests to prove the behavior.

**Why:**
`epic-1a2b-form-ergonomics` and `epic-v151-visual-governance` require strict WCAG adherence. Our previous session identified that `Tooltip` lacked explicit compliance with WCAG 1.4.13 (Content on Hover or Focus). Hover/focus-revealed content must be dismissible without moving the user's focus, allowing them to clear their view without losing their place.

**DESIGN.md sections cited:**
- "2. Accessibility Floor"

**DESIGN.md sections rewritten:**
- Added a strict requirement under "2. WCAG AA Floor" explicitly mandating WCAG 1.4.13 compliance for all tooltips and hover/focus revealed content.
  **Canonical reason:** To formalize the dismissal behavior from an implicit goal into a rigid, system-wide contract.

**Decisions:**
- Attached the keyboard listener directly to the Tooltip's `wrapper` element instead of using a global `window` listener. This aligns with Svelte's atomic event handling and ensures we only intercept `Escape` when the user is explicitly focused on the tooltip's trigger.
- Added `event.stopPropagation()` to prevent the `Escape` keypress from bubbling up and unintentionally closing parent overlays (like `Dialog` or `Drawer`).

**Trade-offs:**
- None. Added minimal bytes for the event listener while significantly improving accessibility and overlay safety.
