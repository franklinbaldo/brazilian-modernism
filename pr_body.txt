What
Implemented the `DatePicker` and `TimePicker` molecular components to support precise, accessible temporal data entry.

Why
The design system needed robust date and time inputs. The aesthetic was built strictly following the 'Curva & Concreto' doctrine: the input fields leverage curved geometry (`--r-1`) for interaction, while the popovers utilize sharp, rigid geometry (`0px` border-radius, pure block structure) for static data presentation.

Decisions
- **Svelte 5 Runes**: Built using `$props`, `$state`, `$derived`, and `$effect`.
- **Timezone Parsing**: Explicitly avoided `new Date('YYYY-MM-DD')` to prevent the UTC midnight evaluation bug. Dates are manually parsed via `split('-')` and instantiated with local timezone arguments.
- **Workflow**: Updated `time-picker.feature` to reflect the decision to leave the time popover open after individual selections (hours/minutes), allowing fluid compound actions rather than jarring closures.
- **Documentation**: Svelte 5 snippets inside Astro MDX threw Acorn parser errors; adjusted MDX documentation to avoid inline snippet definitions while still proving the FormField context works.

Trade-offs
- Rejected relying on native `<input type="date">` and `<input type="time">` because they cannot be reliably styled across browsers, which breaks the doctrine's strict aesthetic constraints. Custom popovers were necessary to enforce "concreto" geometry.

Test evidence
- Vitest BDD tests pass (85 total tests across the suite).
- Pre-commit typecheck and Astro build ran perfectly.
- A11y compiler warnings resolved (removed invalid `aria-expanded` from textboxes and `onkeydown` from non-interactive `div`s).
