What
Implemented the `Tooltip` molecule component to provide context-sensitive, non-essential supplementary information.

Why
Overlay components like tooltips are fundamental for keyboard and visual accessibility, allowing us to describe UI elements without cluttering the screen or hijacking focus. This directly addresses the "a11y" and "micro-interactions" themes in `DESIGN.md`.

Decisions
- **Svelte 5 Runes**: Built using `$props` and `$state` to natively react to interactions.
- **WAI-ARIA**: Manages `aria-describedby` automatically by generating a stable `tooltipId` and injecting it onto the trigger wrapper when open. The tooltip node uses `role="tooltip"`. We use Svelte 5 Snippets `{#snippet children(props)}` to allow spreading the aria-describedby attribute directly onto the semantic focusable target (button, link, etc.) to ensure screen readers read it correctly on focus.
- **Curva & Concreto**: Since a tooltip presents static context data, it relies exclusively on the "Concreto" doctrine. The popup has a `border-radius` of `0px` with `--concreto-90` background.

Trade-offs
- Used absolute positioning relative to a wrapper div (`display: inline-flex`). While a library like Floating UI could handle edge collisions dynamically, keeping it zero-dependency and relying on CSS mapping to `.cobogo-tooltip-popup--{position}` is much lighter and aligns with the strict `<5KB` rule. Developers can choose `top`, `bottom`, `left`, or `right`.
