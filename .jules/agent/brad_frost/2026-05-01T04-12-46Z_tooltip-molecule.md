## 3. THIS SESSION
- **What I did:** Built the `Tooltip` molecule. Created `Tooltip.svelte` and a test wrapper `Tooltip.test.svelte`. Used Svelte 5 `$props` and `$state` to manage the hover/focus interaction natively. Implemented absolute CSS positioning with an arrow for top, bottom, left, and right placements without external dependencies. Wrote BDD tests verifying a11y behaviors like `aria-describedby` dynamic injection.
- **Why:** To fulfill the system's need for non-essential supplementary information overlays. This directly tackles the "micro-interactions" and "a11y" theme requirements while expanding the "Curva & Concreto" doctrine.
- **Decisions made:** Avoided a heavy library like Floating UI to keep the component weight negligible (<5KB constraint). Instead, provided 4 static `position` props mapping to pure CSS positioning logic. Wrapped the trigger snippet in a generic `<div class="cobogo-tooltip-wrapper">` to intercept events (`mouseenter`, `mouseleave`, `focusin`, `focusout`) transparently, eliminating the need for developers to manually bind interactions to their child triggers.
- **Citations to DESIGN.md:** Applied the "Curva & Concreto" doctrine for static data. Tooltips present raw, un-actionable information. Therefore, the popup box has `border-radius: 0` and a solid `--concreto-90` background. Applied "WCAG AA Floor" by automatically setting `role="tooltip"` and toggling `aria-describedby` on the wrapper.

## 6. NOTES TO FUTURE-ME
Svelte 5 testing of transitions with `@testing-library/svelte` can be tricky because the JSDOM environment does not resolve CSS transitions accurately in real time, causing `queryByRole` to sometimes find elements that logically should be unmounted. Advancing timers (`vi.advanceTimersByTime`) and adding small `setTimeout` buffers in the component for `blur`/`mouseleave` makes the testing environment reliable without sacrificing UX.
