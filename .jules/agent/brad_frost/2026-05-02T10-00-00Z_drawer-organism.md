## 3. THIS SESSION
- **What I did:** Built the `Drawer` organism. Created `Drawer.svelte`, `DrawerHeader.svelte`, `DrawerBody.svelte`, and `DrawerFooter.svelte`. Implemented it as a lateral overlay component using the native HTML `<dialog>` element to handle focus trapping, backdrop, and accessibility naturally. Handled CSS transitions with Svelte's `fly` to slide it in from the right or left.
- **Why:** The system needed an overlay component that breaks out of the document flow for dense lateral interactions, typically used for mobile navigation or complex side panels that don't warrant a full modal overlay but still require focus isolation.
- **Decisions made:** Adopted strict composition over configuration. `Drawer` controls the state (`open` bindable) and provides an implicit `dialogClose` context to its children (like the `DrawerHeader` close button). This mirrors the HTML5 approach and allows the developer to compose the Drawer's internals flexibly without dealing with massive configuration objects. Added defensive checks around Svelte Context getters to prevent SSR crashes, a lesson carried over from the Tabs component.
- **Alternatives rejected:** Rejected building a custom focus trap and backdrop overlay with generic `<div>`s in favor of `<dialog>`, which browsers now support fully and provides built-in accessibility guarantees (e.g., `aria-modal="true"`, native `Escape` key handling).
- **Citations to DESIGN.md:** Enforced the "Curva & Concreto" doctrine. The Drawer itself does not have a border-radius where it touches the screen edge but applies `--r-1` exclusively on the side facing the center of the viewport (the interior edge). This conveys that it is an interactive layer sliding over the concrete data below, anchored to the edge. Used standard tokens (`--papel-00` for body, `--papel-10` for footer) and `--shadow-2` for elevation.

## 6. NOTES TO FUTURE-ME
Using `<dialog>` simplifies focus management significantly, but handling the Svelte `$bindable(open)` state requires careful interception. Do not let `<dialog>` close itself naturally without also updating the Svelte state prop (hence why we `preventDefault()` on the `Escape` key and handle it manually, or sync with `$effect`). Vitest JSDOM doesn't implement `showModal` or `close` for `<dialog>`, so they must be mocked in `beforeEach` to simulate native behavior for tests.
